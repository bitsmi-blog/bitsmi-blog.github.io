Focus: High-level organization of clean, maintainable classes in object-oriented systems

📌 Executive Summary
While previous chapters focused on writing clean functions, this chapter zooms out to the class level, emphasizing encapsulation, responsibility assignment, and organization for change. The key messages:
Classes should be small, cohesive, and encapsulate a single responsibility.


Class boundaries should reflect clear responsibilities, support the SRP, OCP, and DIP.


Favor many small, focused classes over a few large, multipurpose ones.


Isolate volatile or change-prone behavior into separate abstractions to reduce impact.


Refactor large methods/functions into cohesive units with minimal instance variable sharing.



🧱 Structure of a Clean Class
Following Java conventions, the internal structure of a clean class is organized as:
public class Example {
// 1. Public static constants
public static final int DEFAULT_SIZE = 10;

    // 2. Private static variables
    private static Logger logger = LoggerFactory.getLogger(Example.class);

    // 3. Private instance variables
    private int size;

    // 4. Public methods
    public int getSize() { return size; }

    // 5. Private helper methods (immediately after the public method that uses them)
    private void validateSize(int newSize) { /*...*/ }
}

🧭 Follow the Stepdown Rule: code should read top-down, like a newspaper.

🧱 Encapsulation
Encapsulation isn't absolute, but it should be preserved unless testing demands otherwise. Use protected or package-private only when necessary.

🔩 The Single Responsibility Principle (SRP)
Definition:
A class should have one and only one reason to change.
❌ Bad Example – God Class:
public class SuperDashboard extends JFrame implements MetaDataUser {
public Component getLastFocusedComponent();
public int getMajorVersionNumber();
public void showProgress(String s);
public MetaObject getSelectedObject();
// ... and ~70 more methods
}

Too many unrelated responsibilities (UI focus, versioning, project state).


Difficult to maintain or extend.


✅ Refactored Example:
public class Version {
public int getMajorVersionNumber();
public int getMinorVersionNumber();
public int getBuildNumber();
}

💡 Heuristic: If you struggle to name a class clearly and concisely, it likely has too many responsibilities.

📐 Cohesion
Definition:
A class is cohesive when its methods operate on the same instance variables.
Example – High Cohesion:
public class Stack {
private int topOfStack = 0;
private List<Integer> elements = new LinkedList<>();

    public void push(int element) {
        topOfStack++;
        elements.add(element);
    }

    public int pop() {
        if (topOfStack == 0) throw new EmptyStackException();
        int result = elements.get(--topOfStack);
        elements.remove(topOfStack);
        return result;
    }

    public int size() {
        return topOfStack;
    }
}

📌 If only a subset of methods use certain variables, consider extracting a new class.

⚒️ Refactoring in Practice – PrintPrimes
❌ Original (Monolithic):
A 100+ line main() function with deeply nested logic and confusing state tracking.
public class PrintPrimes {
public static void main(String[] args) {
//... dense logic, many temp variables
}
}

✅ Refactored Design:
Split by Responsibility
+-------------------+         +-----------------------------+
| PrimePrinter      |  --->   | Handles orchestration and  |
| (main method)     |         | user interface layer        |
+-------------------+         +-----------------------------+

+----------------------------+
| PrimeGenerator             |
| (Static class)             |
| - Generates N primes       |
+----------------------------+

+----------------------------+
| RowColumnPagePrinter       |
| - Formats and prints table |
+----------------------------+

Sample:
int[] primes = PrimeGenerator.generate(NUMBER_OF_PRIMES);
RowColumnPagePrinter printer = new RowColumnPagePrinter(...);
printer.print(primes);

🛠️ By refactoring, we’ve increased readability, cohesion, and flexibility.

🧪 Designing for Change
Classes must embrace change by adhering to SRP and OCP.
❌ Violating SRP:
public class Sql {
public String insert(Object[] fields);
public String selectAll();
public String findByKey(...);
public String select(Criteria criteria);
// ...
}

Multiple reasons to change (adding new SQL types, changing formatting logic, etc.)


Changes to one method risk breaking unrelated features


✅ SRP + OCP Friendly Refactor:
abstract class Sql {
public abstract String generate();
}

public class InsertSql extends Sql {
@Override
public String generate() { /* ... */ }
}

public class SelectSql extends Sql {
@Override
public String generate() { /* ... */ }
}

➕ Add new types (like UpdateSql) without touching existing code → Open for extension, Closed for modification (OCP)

🔌 Dependency Inversion Principle (DIP)
❌ Direct dependency:
public class Portfolio {
private TokyoStockExchange exchange; // Tight coupling
}

✅ Inverted dependency:
public interface StockExchange {
Money currentPrice(String symbol);
}

public class Portfolio {
private final StockExchange exchange;
public Portfolio(StockExchange exchange) {
this.exchange = exchange;
}
}

✅ Enables mocking during tests
✅ Reduces change ripple
✅ Improves flexibility
Test Example:
@Test
public void testMSFTPortfolioValue() {
FixedStockExchangeStub exchange = new FixedStockExchangeStub();
exchange.fix("MSFT", 100);
Portfolio portfolio = new Portfolio(exchange);
portfolio.add(5, "MSFT");
assertEquals(500, portfolio.value());
}


🔁 Summary – Class Design Checklist
Principle
Goal
SRP
Each class has one reason to change
Cohesion
Methods share and depend on the same data
OCP
Extend without modifying existing code
DIP
Depend on abstractions, not concrete implementations
Encapsulation
Keep internals private unless justified (e.g., testing)
Small Size
Favor small, clearly defined classes
