🧠 Key Idea
“Systems must be clean too.”
Clean architecture and separation of concerns are as critical at the system level as clean code is at the class/method level. Poor system design compromises testability, scalability, and developer productivity.

🏗️ 1. Separate System Construction from Use
✳️ Problem:
Runtime logic often gets coupled with object creation logic, violating Single Responsibility Principle.
❌ Bad Example: Lazy Initialization (Hard-coding dependency)
public Service getService() {
if (service == null)
service = new MyServiceImpl(); // Tight coupling
return service;
}

✅ Solution:
Perform object construction and wiring only in the main() method (or modules it calls).


Keep the rest of the application decoupled from construction logic.


✅ Good Example:
public static void main(String[] args) {
Service service = new MyServiceImpl();
Application app = new Application(service);
app.run();
}


🏭 2. Factories & Dependency Injection (DI)
🎯 Goal:
Let application code control when objects are created without being tied to how they are created.
🏭 Abstract Factory Pattern
public interface LineItemFactory {
LineItem createLineItem(String productId, int qty);
}

public class OrderProcessor {
private final LineItemFactory factory;

    public OrderProcessor(LineItemFactory factory) {
        this.factory = factory;
    }

    public void addItem(String productId, int qty) {
        LineItem item = factory.createLineItem(productId, qty);
        // ...
    }
}

🧰 Dependency Injection (DI)
An external container wires dependencies together.


Promotes Inversion of Control and Testability.


public class OrderService {
private final PaymentGateway gateway;

    public OrderService(PaymentGateway gateway) {
        this.gateway = gateway;
    }
}

🛠 Spring Example (XML Configuration):
<bean id="paymentGateway" class="com.example.PayPalGateway"/>
<bean id="orderService" class="com.example.OrderService">
<constructor-arg ref="paymentGateway"/>
</bean>


🌆 3. Scaling Systems Like Cities
Cities evolve incrementally; so should software systems.
Don’t attempt Big Design Up Front (BDUF)


Implement just enough for today’s stories, and refactor for future ones.


Modularize to decouple scaling from initial implementation.



🧱 4. EJB2: A Case of Invasive Architecture ❌
Problems with EJB2:
Required verbose XML descriptors


Tight coupling to containers


Difficult to test


Required DTOs and repetitive code


public abstract class Bank implements EntityBean {
public abstract String getStreetAddr1();
// Many methods and boilerplate
}


✅ 5. Modern Approaches: Spring, POJOs & AOP
✅ Clean, Decoupled POJOs (Spring/EJB3):
@Entity
public class Bank {
@Id @GeneratedValue
private int id;

    @Embedded
    private Address address;

    @OneToMany(mappedBy="bank")
    private List<Account> accounts;

    // Getters, setters
}

☕ Spring DI (POJO + Configuration):
<bean id="bankDAO" class="BankDataAccessObject" p:dataSource-ref="dataSource"/>
<bean id="bank" class="Bank" p:bankDAO-ref="bankDAO"/>

🧩 Each bean wraps the other like a Russian Doll of decorators


🪞 6. Cross-Cutting Concerns & AOP
Cross-Cutting Concerns:
Persistence


Logging


Transactions


Security


🔧 Java Proxy Example:
Bank bank = (Bank) Proxy.newProxyInstance(
Bank.class.getClassLoader(),
new Class[] { Bank.class },
new BankProxyHandler(new BankImpl()));

Limitations:
Verbose


Hard to scale


Lacks systemic behavior injection


✅ Spring AOP / AspectJ:
Cleaner modularity


Declarative and non-invasive


Maintain POJO simplicity


@Aspect
public class LoggingAspect {
@Before("execution(* com.example.Bank.*(..))")
public void logBefore(JoinPoint joinPoint) {
System.out.println("Logging: " + joinPoint.getSignature().getName());
}
}


📈 7. Test-Drive Your Architecture
If system logic is decoupled using DI and AOP, architecture itself becomes testable.
Enables evolutionary design


Avoids premature commitment


Keeps system open to change



🧠 8. Decision-Making Principles
Decentralized decisions: Push choices to the edges


Just-in-Time decisions: Delay for better information


Use standards only when necessary



🧾 9. Domain-Specific Languages (DSLs)
DSLs help express domain logic in natural, readable forms


Minimize misunderstanding


Examples: JMock, SQL DSLs, or even Spring configuration


mockery.checking(new Expectations() {{
oneOf(order).addItem("item-123", 2);
}});


✅ Conclusion
🔍 Principle
✅ Action
Separate Construction & Use
Use main() or DI container for object creation
Avoid Invasive Architectures
Favor POJOs over container-bound objects
Decouple Cross-Cutting Concerns
Use AOP or declarative configs (Spring, AspectJ)
Design for Evolution
Build simple, testable architectures that scale
Use Standards Wisely
Don’t adopt tech just because it’s a “standard”
Embrace DSLs
Improve domain-code alignment for better understanding

“Systems must be clean too. At all levels of abstraction, the intent should be clear.”
